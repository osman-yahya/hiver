# 🐝 Hiver — Lightweight & AI-Enhanced Server Monitoring

**Hiver** is a modern, ultra-lightweight server and Docker container monitoring system designed for homelabs, self-hosted environments, and internal networks. It provides a beautiful, real-time dashboard to track system resources (CPU, RAM, Disk, Network I/O) and container health — without the massive overhead of traditional enterprise tools.

To take it a step further, Hiver offers an **optional, UI-controlled AI Engine** powered by Ollama. When enabled, it intercepts Docker error logs and translates complex stack traces into human-readable explanations. If the AI is disabled or unreachable, Hiver gracefully falls back to displaying standard raw logs, ensuring your monitoring is **never interrupted**.

---

## 🛠 Architecture & Tech Stack

Hiver is built on a distributed, agent-based architecture prioritizing low resource consumption on target machines and centralized processing on the Mother server.

| Layer | Technology | Responsibility |
|---|---|---|
| **Agent** | Go (Golang) | Statically compiled single-binary. Runs on every monitored server. Reads `/proc` and Docker socket. |
| **Mother (Backend)** | Python / FastAPI | Receives agent payloads, manages WebSockets for real-time UI push, orchestrates the log queue. |
| **Queue** | Redis | Asynchronous task queuing for error log processing. Decouples log ingestion from AI analysis. |
| **Database** | SQLite (default) / PostgreSQL (opt-in) | Persists server registry, metric history, user accounts, alert rules, and processed logs. |
| **UI (Frontend)** | React + Tailwind CSS | Responsive dark-mode dashboard. Real-time updates via WebSocket. |
| **AI Translator** | Ollama (external, opt-in) | Locally-hosted LLM (LLaMA3, Mistral, Phi-3, etc.). Not part of the Hiver compose stack. |

## ✨ Feature Set

### 1. Zero-Friction Agent Deployment

Agents are deployed via a **single Docker command** auto-generated by the UI. Upon first contact, the agent:
1. Pings the Mother server's registration endpoint.
2. Receives a unique `AGENT_TOKEN` for authenticated communication.
3. Begins streaming heartbeat + metric payloads at the configured interval.

You can dynamically adjust the polling rate for all agents — or per-agent — directly from the dashboard with **no container restarts required**.

---

### 2. Real-Time Metrics Collection

The Go Agent collects and transmits the following metrics on every heartbeat:

- **CPU:** Per-core utilization + total average (via `/proc/stat`)
- **Memory:** Total / Used / Free / Cached / Buffers (via `/proc/meminfo`)
- **Disk:** Per-mount usage (GB used, %, inode usage)
- **Network I/O:** Bytes in/out per interface, packet drop rates (via `/proc/net/dev`)
- **Load Averages:** 1m / 5m / 15m system load
- **Uptime:** System uptime in human-readable form
- **Docker Containers:** Per-container CPU %, memory usage, status, restart count, and image name
- **Process Snapshot (opt-in):** Top N processes by CPU/RAM, useful for anomaly hunting

---

### 3. Metrics History & Sparklines

Hiver stores metric snapshots in the database at configurable retention intervals. This enables:

- **Sparkline graphs** on every server card (CPU/RAM trend over last hour/day)
- **Full history charts** on the detailed server view (zoomable time range: 1h / 6h / 24h / 7d)
- **Peak usage tracking** — record the all-time highest CPU/RAM reading per server
- **Retention policy** — configurable in the Admin Panel (e.g., keep hourly snapshots for 7 days, daily for 90 days)

---

### 4. Smart Log Filtering & Redis Queuing

Instead of streaming gigabytes of useless `INFO` logs, the Go Agent filters at the source. It transmits only lines matching configurable keywords (default: `error`, `panic`, `exception`, `fatal`, `critical`, `killed`, `oom`).

The Mother server receives these logs and:
1. Immediately pushes them to the **Redis queue** (prevents blocking).
2. A background worker pulls from the queue and either routes to the AI engine or saves the raw log directly.

**Custom keyword lists** can be added per-server group from the Admin Panel.

---

### 5. Graceful AI Integration (UI-Controlled)

Hiver does not force AI usage. Configuring it requires **zero file editing or container rebuilds**.

#### Ollama Settings (Admin Panel → AI Engine)

| Setting | Description | Example |
|---|---|---|
| **Enable AI Analysis** | Master toggle. Disabling this bypasses the AI worker entirely. | `ON / OFF` |
| **Ollama API URL** | The base URL of your Ollama server. | `http://192.168.1.50:11434` |
| **Model Name** | The Ollama model to use for analysis. | `llama3`, `mistral`, `phi3` |
| **Request Timeout (s)** | Max seconds to wait for an Ollama response before falling back. | `30` |
| **System Prompt** | Custom instruction prepended to every log analysis request. Tailor Ollama's persona and verbosity. | `"You are a concise DevOps assistant..."` |
| **Max Log Length (chars)** | Truncation limit for logs sent to Ollama. Prevents exceeding context windows. | `4000` |
| **Concurrent Workers** | Number of parallel Ollama requests the Mother will make. | `2` |
| **Test Connection** | One-click button to ping the Ollama `/api/tags` endpoint and list available models. | ✅ |

#### How It Works
1. A background worker **pulls** an error log from the Redis queue.
2. It **constructs a prompt**: system prompt + log context + the raw error text.
3. It **POSTs** to `{OLLAMA_URL}/api/generate` with the configured model.
4. The plain-English explanation is saved to the database alongside the original raw log.
5. The UI receives a WebSocket push and renders the explanation in the Error Logs view.

#### Graceful Degradation
If AI is toggled off, the Ollama server is unreachable, or the request times out — the worker catches the exception and saves the **raw log string** directly. Core monitoring is **never interrupted**.

### 6. Smart Alerting System

Define threshold-based alert rules from the UI with **zero config files**:

- **Metric Alerts:** Trigger when CPU > X%, RAM > Y%, Disk > Z% for more than N minutes.
- **Container Alerts:** Trigger when a container exits unexpectedly, restarts more than N times in M minutes, or transitions to `unhealthy` status.
- **Heartbeat Alerts:** Trigger when an agent misses N consecutive heartbeats (server is likely DOWN).
- **AI-Augmented Alerts (opt-in):** When an alert fires, optionally ask Ollama to summarize what might have caused it based on recent logs.

#### Notification Channels (pluggable)

| Channel | Notes |
|---|---|
| **Webhook** | Send a JSON POST to any URL (Discord, Slack, n8n, etc.) |
| **Email (SMTP)** | Configure SMTP credentials in the Admin Panel |
| **Gotify / ntfy** | Push to self-hosted notification servers |
| **Telegram Bot** | Bot token + chat ID configured in the UI |
| **In-App** | Always on. Alerts appear in the notification bell and Alert History page. |

Alert rules support **cooldown periods** to prevent notification floods.

---

### 7. Heartbeat & Downtime Detection

Agents send a lightweight heartbeat payload on every poll cycle. The Mother server tracks the `last_seen` timestamp per agent. If an agent misses **N consecutive heartbeats** (configurable), the server is immediately flagged as:

- **🟡 Degraded** — Missing 1–2 heartbeats (network blip).
- **🔴 DOWN** — Missing N+ heartbeats (host crash / network failure).

Downtime events are logged with start/end timestamps and duration, visible in the **Incident History** view.

---

### 8. Role-Based Access Control (RBAC)

Hiver supports multiple user roles manageable from the Admin Panel:

| Role | Permissions |
|---|---|
| **Admin** | Full access: manage servers, users, alert rules, AI settings, view all logs. |
| **Operator** | View dashboard, view logs, acknowledge alerts. Cannot change settings or manage users. |
| **Kiosk** | Read-only access to Kiosk Mode only. Designed for dedicated monitors. |

Authentication is session-based (JWT). Initial admin credentials are set on first run via environment variable or interactive setup wizard.

---

### 9. Server Groups & Tags

Organize monitored servers into logical **groups** (e.g., `Production`, `Staging`, `Storage`):

- Filter the dashboard by group.
- Apply alert rules to all servers in a group at once.
- Assign group-specific log keyword filters.
- Display group-level aggregate stats (total CPU, total RAM across group).

Servers can also have free-form **tags** (e.g., `nginx`, `database`, `arm64`) for search and filtering.

---

### 10. Kiosk Mode

A distraction-free, full-screen UI designed for wall-mounted or dedicated monitoring screens:

- Servers displayed as bold, color-coded status cards.
- Auto-rotates between server groups if more servers than fit in view.
- No navigation elements or clickable controls (read-only).
- Configurable refresh rate (independent of global poll interval).
- Time & date overlay.

Status colors:
- 🟢 **Green** — Stable (all metrics below warning thresholds, no errors).
- 🟠 **Orange** — Warning (high resource usage, or container restart detected).
- 🔴 **Red** — Critical (server DOWN, or active unacknowledged critical alert).
- ⚫ **Grey** — Unknown / Agent not yet registered.

---

### 11. Audit Log

Every significant action taken within Hiver is recorded:

- User login / logout events with IP and timestamp.
- Configuration changes (who changed what, old vs. new value).
- Server additions / removals.
- Alert rule creation / modification / deletion.
- AI settings changes.

Audit logs are append-only and viewable by Admins only.

---

### 12. Import / Export & Backup

- **Export:** Download the entire server registry, alert rules, and settings as a single JSON file for backup or migration.
- **Import:** Restore from a JSON backup on a fresh Hiver installation.
- **Scheduled DB Backup (opt-in):** Auto-dump the database to a configurable path on a cron schedule.

## 🔧 Configuration Reference

All settings are persisted in the database and managed via the UI. A minimal `.env` file is only needed for first-run bootstrap:

```env
# --- Mother Server Bootstrap ---
HIVER_SECRET_KEY=change_me_to_a_long_random_string
HIVER_ADMIN_USER=admin
HIVER_ADMIN_PASSWORD=changeme

# --- Database (default: SQLite) ---
DATABASE_URL=sqlite:///./hiver.db
# For PostgreSQL: DATABASE_URL=postgresql://user:pass@host:5432/hiver

# --- Redis ---
REDIS_URL=redis://redis:6379/0

# --- Server ---
HIVER_PORT=8000
HIVER_HOST=0.0.0.0
```

> All Ollama settings (URL, model, prompt, timeouts, etc.) are configured **exclusively from the Admin Panel UI** after first login. There are no Ollama-related environment variables to set.

## 🐳 Deployment Overview

### Mother Stack (`docker-compose.yml`)

```
hiver-mother   (FastAPI backend + static React build)
hiver-redis    (Redis message broker)
hiver-db       (SQLite volume or external PostgreSQL)
```

### Go Agent (one per monitored server)

```bash
docker run -d \
  --name hiver-agent \
  --restart unless-stopped \
  -e MOTHER_URL=http://<your-mother-ip>:8000 \
  -e SERVER_LABEL=my-server-name \
  -v /var/run/docker.sock:/var/run/docker.sock:ro \
  -v /proc:/host/proc:ro \
  --pid host \
  osmanyahyaakinci/hiver-agent:latest
```

The UI auto-generates this command (with the correct `MOTHER_URL` and unique `SERVER_LABEL`) when you click **"Add Server"**.
